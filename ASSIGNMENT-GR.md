## Εργασία Εργαστηρίου Λειτουργικών Συστημάτων - Ανάλυση και Επεξεργασία Συμβάντων Συστήματος σε UNIX Περιβάλλον

Η Διεύθυνση Πληροφορικής ενός οργανισμού θέλει μια μικρή UNIX-based υποδομή
παρακολούθησης συμβάντων. Καλείστε να υλοποιήσετε βήμα-βήμα ένα mini-system που:
1. Συλλέγει αρχεία καταγραφής (logs)
2. Φιλτράρει τα περιεχόμενα με regex
3. Συνδυάζει εντολές με pipes & redirections
4. Διαχειρίζεται διεργασίες για την εκτέλεση εργασιών
5. Καλεί βοηθητικά προγράμματα γραμμένα σε C
6. Αυτοματοποιεί τη ροή μέσω shell scripts
7. Χρησιμοποιεί threads για παράλληλη επεξεργασία δεδομένων

---

## ΜΕΡΟΣ Α – Προετοιμασία περιβάλλοντος & βασικές εντολές (0.5 μονάδες)

Δημιουργείτε μια δομή καταλόγων για το σύστημα καταγραφής:
```
monitor/
 raw/
 processed/
 reports 
```
1. Με μία εντολή δημιουργήστε τη δομή
2. Στον φάκελο `monitor/raw/` δημιουργήστε τρία (3) αρχεία `.log`:
- `system.log`
- `network.log`
- `security.log`
3. Σε καθένα προσθέστε ενδεικτικές γραμμές (με `echo >>`), π.χ. warnings, errors,
   timestamps κτλ, που να εμπεριέχουν:
- Γραμμές που ξεκινούν με ημερομηνία (pattern: `YYYY-MM-DD`)
- Γραμμές που περιέχουν λέξεις ERROR, FAILED, CRITICAL
- Γραμμές που περιέχουν IPv4 διεύθυνση
4. Εμφανίστε αναλυτική λίστα των αρχείων και μετρήστε πόσες γραμμές συνολικά
   υπάρχουν στα τρία (3) logs

---

## ΜΕΡΟΣ Β – Φιλτράρισμα logs με Regular Expressions (0.5 μονάδες)

Θέλουμε να εντοπίσουμε συμβάντα που απαιτούν προσοχή.

1. Χρησιμοποιώντας `grep` με regex, να εντοπίσετε σε όλα τα logs:
- Γραμμές που ξεκινούν με ημερομηνία (pattern: `YYYY-MM-DD`)
- Γραμμές που περιέχουν είτε τις λέξεις ERROR, FAILED, CRITICAL
- Γραμμές που περιέχουν IPv4 διεύθυνση
2. Το αποτέλεσμα να αποθηκευτεί στο αρχείο: `monitor/processed/alerts.raw`
3. Αφαιρέστε διπλότυπες γραμμές και ταξινομήστε τες, δημιουργώντας το αρχείο:
   `monitor/processed/alerts.sorted`

---

## ΜΕΡΟΣ Γ – Συνδυασμός Pipes & Redirection για δημιουργία αναφοράς (0.5 μονάδες)

Χρησιμοποιώντας μόνο μία pipeline εντολή:
1. Εξαγάγετε από το αρχείο `monitor/processed/alerts.sorted`
- πόσα total alerts υπάρχουν
- πόσα περιέχουν τη λέξη ERROR
- πόσα αφορούν συγκεκριμένο IP range (π.χ. ξεκινά με `192.168.*`)
2. Η εντολή πρέπει να παράγει αρχείο: `monitor/reports/daily_summary.txt` με περιεχόμενο όπως:
   Total Alerts: X, Errors: Y, Local Network Events: Z

---

## ΜΕΡΟΣ Δ – Διαχείριση Διεργασιών (0.4 μονάδες)

Θα χρειαστεί να παρακολουθήσετε διεργασίες που σχετίζονται με το σύστημά σας.

1. Εκτελέστε μια εντολή που "τρέχει στο παρασκήνιο (background)" και γράφει περιοδικά
   timestamps σε ένα αρχείο `raw/timestamps.log`
2. Με ps και grep εντοπίστε το process ID (PID) της διεργασίας
3. Αλλάξτε την προτεραιότητα της διεργασίας (`nice` ή `renice`)
4. Τερματίστε την πρώτα με `-TERM` και, αν χρειαστεί, με `-KILL`
5. Καταγράψτε τα `PID` και τι συνέβη

---

## ΜΕΡΟΣ Ε – Υλοποίηση προγράμματος C για ανάλυση logs (0.6 μονάδες)

Θα χρειαστεί να γράψετε πρόγραμμα C με τίτλο: `analyze.c`. Το πρόγραμμα:
1. Λαμβάνει ως όρισμα ένα όνομα αρχείου `.log`.
2. Προσπαθεί να το ανοίξει με `open()`.
3. Αν αποτύχει → χρησιμοποιεί `errno`, `perror()` για να εμφανίσει μήνυμα λάθους
4. Αν ανοίξει, το διαβάζει γραμμή-γραμμή και μετρά:
- πλήθος γραμμών
- πόσες περιέχουν τη λέξη ERROR
- πόσες περιέχουν αριθμό
5. Επιστρέφει exit code:
- `0` → επιτυχία
- `1` → σφάλμα ανοίγματος
- `2` → κενό αρχείο

Το εκτελέσιμο θα ονομάζεται: `analyze_log`

---

## ΜΕΡΟΣ ΣΤ – Shell Script που αυτοματοποιεί όλο το σύστημα (0.7 μονάδες)

Θα χρειαστεί να γράψετε το script με τίτλο: `run_monitor.sh`. Το script πρέπει:
1. Να δέχεται ως όρισμα έναν φάκελο logs (π.χ. `monitor/raw/`)
2. Να ελέγχει ότι δόθηκε όρισμα (positional parameters)
3. Να ελέγχει αν ο φάκελος υπάρχει - αλλιώς να τερματίζει με μήνυμα
4. Να εμφανίζει το όνομα του script και τον συνολικό αριθμό ορισμάτων
5. Να καλεί το πρόγραμμα `analyze_log` για κάθε αρχείο στο directory
6. Να συγκεντρώνει όλα τα αποτελέσματα σε ένα νέο αρχείο: `monitor/reports/full_report.txt`
7. Να χρησιμοποιεί:
- `for` loops
- `case` (για categorization, π.χ. system/network/security log)
- `while` loop αν επιθυμείς επαναληπτική εκτέλεση
- `IFS` για ασφαλή διάσπαση γραμμών

---

## ΜΕΡΟΣ Ζ – Threads για παράλληλη ανάλυση logs (0.8 μονάδες)

Θα πρέπει να επιταχύνετε το σύστημα με thread μέσω του προγράμματος: `parallel_analyze.c`

1. Το πρόγραμμα δέχεται περισσότερα από ένα αρχεία `.log` ως ορίσματα
2. Δημιουργεί ένα thread για κάθε αρχείο, όπου κάθε thread:
- ανοίγει το αρχείο
- μετρά πλήθος γραμμών & errors (όπως στο analyze.c)
- επιστρέφει τα αποτελέσματα στη main (μέσω struct)
3. Η `main()` κάνει `pthread_join()` για όλα τα threads
4. Εκτυπώνει συνολική αναφορά στη μορφή:
```
File: X.log | Lines: L | Errors: E
                ...
          TOTAL LINES: ...
          TOTAL ERRORS: ...
```

## Προσοχή

- Μην χρησιμοποιήσετε κοινή μεταβλητή χωρίς προστασία
- Πρέπει να δοθεί ξεχωριστή struct σε κάθε thread

- Η εργασία είναι <u>**ατομική**</u> και βαθμολογείται με <u>**4 μονάδες**</u>
- Η εργασία θα <u>**παραδοθεί**</u> μέσω [**eClass**](https://eclass.hua.gr/), σε ημερομηνία που θα ανακοινωθεί μελλοντικά, στην

  αρχείου. Ο φάκελος του συμπιεσμένου αρχείου θα πρέπει να περιέχει:
  - Όλες τις **απαντήσεις του project** (χωρισμένο ανά ΜΕΡΟΣ) συνδυαστικά με την
    τεκμηρίωσή τους αναφορικά με τις εντολές, τον τρόπο σκέψης σας και τον τρόπο
    επίλυσης του κάθε ερωτήματος, συνοδευόμενο από σχετικά screenshots από την
    εκτέλεση των εντολών από το terminal σας
  - Το script `run_monitor.sh`
  - Τα προγράμματα C: `analyze.c` και `parallel_analyze.c`
  - Ένα μικρό `README` αρχείο με οδηγίες εκτέλεσης
- Τα εκτελέσιμα αρχεία θα πρέπει να εκτελούνται σε λειτουργικό σύστημα Linux
- Οι διδάσκοντες θα αποφασίσουν ποιοι θα κληθούν για προφορική εξέταση, οι οποίοι θα
  ενημερωθούν μέσω [**eClass**](https://eclass.hua.gr/)/email. Τα κριτήρια επιλογής θα είναι: (α) Μη τεκμηριωμένη
  εργασία, (β) Ελλιπής αναφορά, ή (γ) Μη ύπαρξη σχολίων/μηνυμάτων
- Σε περίπτωση που παρατηρηθεί πανομοιότυπος κώδικας (αντιγραφή), η εργασία θα
  μηδενιστεί αυτόματα
